# TS-Go-Accepted Baseline Overlays

TSTS is a mechanical 1:1 port of the pinned TS-Go compiler (`packages/tsts/_vendor/typescript-go`).
The exact-baseline gate of the tsgo-suite compares TSTS output against the Strada-generated
reference baselines vendored under `_submodules/TypeScript/tests/baselines/reference`.

For a small number of artifacts, the **pinned TS-Go compiler itself demonstrably diverges from
the Strada baselines** (chiefly isolated-declarations emit and source-map spans — surface Strada
exercises through its transpile suite but TS-Go never runs). Forcing TSTS to match Strada there
would mean deliberately diverging from TS-Go, breaking the mechanical-port invariant. This is the
same class of drift Microsoft tracks upstream with their `submoduleAccepted`/`submoduleTriaged`
baseline trees for the suites TS-Go does run.

The files in this directory capture **pinned TS-Go's actual output** for exactly the divergent
sections. The runner (`run.mjs`) compares those sections against the overlay and everything else
against the Strada baseline, and reports every overlay use in the run summary
(`Exact-baseline tsgo-accepted sections`).

## Two kinds of overlay section

1. **Content overlay** (`sections` in the manifest): pinned TS-Go emits the section, but with
   content different from Strada. The overlay records pinned's content; the runner compares that
   section against the overlay instead of Strada.
2. **Absent section** (`absentSections` in the manifest): pinned TS-Go emits **no file** for a
   section the Strada baseline still has — chiefly a `.d.ts`/`.d.ts.map` that the new pin blocks
   under `--isolatedDeclarations`. The overlay records an explicit absence marker
   (`TSGO_ACCEPTED_ABSENT_MARKER`); the runner deletes that section from the expected outputs so
   the intentional non-emission is not reported as a missing baseline output. TSTS faithfully
   blocks the same emit, so the gate passes. `transpileWorker` (`src/services/transpile.ts`)
   returns empty output plus diagnostics for a blocked declaration emit (mirroring tsc's
   `transpileModule`), and the runner skips writing the (empty) file.

Because transpile diagnostics live in the `.d.ts` baseline's `Diagnostics reported` section rather
than a `.errors.txt`, in exact mode the runner derives a transpile case's `expectedErrors` from the
**post-overlay** `Diagnostics reported` (`evaluateExactBaselines().expectedDiagnosticsPresent`), not
from `.errors.txt`.

## Rules

- Overlays are generated by `capture-tsgo-accepted.mjs` from a real pinned-TS-Go binary run.
  Never hand-edit an overlay. Never derive an overlay from TSTS output.
- A content-overlay section must differ from the corresponding Strada section; the capture tool and
  the runner both fail on stale (identical) overlays.
- An `absentSections` entry must exist in the Strada baseline **and** be absent from real pinned
  TS-Go output; the capture tool fails if pinned actually emitted it.
- Overlay sections may only override/mark-absent sections that exist in the Strada baseline.
- Every overlay file/section pair is declared in `manifest.json` (`sections` + `absentSections`).

## Regenerating

```sh
cd packages/tsts/_vendor/typescript-go
go build -o /tmp/tsgo ./cmd/tsgo
cd ../../../..
node packages/tsts/tools/tsgo-suite/capture-tsgo-accepted.mjs --tsgo /tmp/tsgo
```

Regeneration is required whenever the pinned TS-Go revision changes.
